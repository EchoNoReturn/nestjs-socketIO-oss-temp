## JWT authentication

### Required environment variables

The service uses JWT for HTTP and WebSocket authentication.

- `JWT_SECRET` (required): secret used to sign/verify access tokens
- `JWT_EXPIRES_IN` (optional, default `1h`): token lifetime
  - Supported formats: seconds (e.g. `3600`) or short duration (e.g. `15m`, `1h`, `7d`)

Example: copy `.env.example` to `.env` and adjust values.

```bash
cp .env.example .env
```

Note: `.env` is gitignored; do not commit secrets.

### HTTP flow

1. Login to get token

- `POST /api/users/login`
- Body: `{ "account": "username-or-email-or-phone", "password": "..." }`
- Response includes: `access_token`

2. Call protected APIs with Bearer token

- Header: `Authorization: Bearer <access_token>`
- Example protected endpoint: `GET /api/users`

Server-side verification:

- `JwtAuthGuard` enforces auth
- `JwtStrategy` verifies signature/expiration and checks user exists (and not soft-deleted)

### Global guard + JumpAuth

Auth is enforced globally via a global guard.

- Default: endpoints require JWT
- Mark an endpoint as public: `@JumpAuth()`
- Require permissions: `@JumpAuth(['perm:a', 'perm:b'])`

### WebSocket flow

For guarded message handlers (e.g. `GameGateway` `message` event), provide token during the handshake using any of:

- `handshake.auth.token = "<token>"` (supports optional `Bearer ` prefix)
- `handshake.headers.authorization = "Bearer <token>"`
- `handshake.query.token = "<token>"`

Server-side verification:

- `WsJwtGuard` verifies token and checks user exists (and not soft-deleted)
- On success, attaches `{ id, username }` to `client.data.user`

## Single-session (single instance)

This service is currently designed with **single-instance** WebSocket kick-out behavior.

### How it works

- On login, the server generates a `sid` (session id) and stores it in Redis under `auth:sid:<userId>`.
- The access token includes `{ sub, sid }`.
- For every HTTP request and WebSocket message, the server verifies that the token `sid` matches Redis.
  - If a second device logs in, Redis `sid` is replaced, and older tokens become invalid immediately.

For the single-instance WebSocket kick-out, sockets store their `sid` in `socket.data.sid`, and on login the server disconnects sockets whose `sid` does not match the newly issued token.

Additionally (single-instance only), the server will disconnect existing WebSocket connections for that user on new login and emit:

- `force_logout` event with `{ reason: 'logged_in_elsewhere' }`

### Configuration

Primary configuration examples live in `config/config.yaml.example`.

- `auth.singleSession` (default: `true`): enables Redis-backed single-session validation
- `auth.wsKickOnLogin` (default: `true`): disconnect old WebSocket connections on new login (single-instance only)

### Moving to multi-instance

The Redis `sid` check works across instances, so old tokens still become invalid. However, **immediate WebSocket disconnect** requires extra work:

1. Add a cross-instance mechanism to target sockets:
   - Option A: socket.io Redis adapter (recommended)
   - Option B: Redis pub/sub channel (broadcast `kick(userId)`; each instance disconnects its local sockets)
2. Ensure the gateway handshake auth middleware runs on all instances.

## Snowflake user IDs

User primary keys are generated by the application using Snowflake IDs (`@akashrajpurohit/snowflake-id`), not auto-increment.

### Configuration

- `config.id.snowflake.workerId` (0-1023): required for multi-instance, must be unique per instance
- `config.id.snowflake.epoch` (ms): optional

Environment variable overrides:

- `SNOWFLAKE_WORKER_ID`: overrides workerId
- `SNOWFLAKE_MACHINE_ID`: overrides workerId (kept for compatibility)
- `SNOWFLAKE_EPOCH`: overrides epoch

### Multi-instance notes

Snowflake IDs include a per-instance worker id. If two instances share the same worker id and generate IDs at the same time, collisions are possible.

Recommended ways to set a unique worker id:

1. Kubernetes StatefulSet ordinal: set `SNOWFLAKE_WORKER_ID` from pod name index.
2. Configuration management: assign a fixed `workerId` per instance via config/secret.
3. Process-based fallback (not recommended for clusters): `process.pid % 1024`.

## SMTP mail

This service includes SMTP mail infrastructure using `@nestjs-modules/mailer` + Handlebars templates.

### Configuration

Mail is disabled by default. Configure it in `config/config.yaml` (see `config/config.yaml.example`):

- `mail.enabled`: set to `true` to enable sending
- `mail.from`: default `From` header
- `mail.templatesDir`: template root directory (default `config/templates/mail`)
- `mail.smtp.host`, `mail.smtp.port`, `mail.smtp.secure`, `mail.smtp.auth.user`, `mail.smtp.auth.pass`

### Templates

Templates are individual `.hbs` files in `config/templates/mail/`:

- `<templateName>-subject.hbs` (required)
- `<templateName>-text.hbs` (optional)
- `<templateName>-html.hbs` (optional)

Example: for a "welcome" template:

- `config/templates/mail/welcome-subject.hbs`
- `config/templates/mail/welcome-text.hbs`
- `config/templates/mail/welcome-html.hbs`

### Usage

Inject `MailService` and call either:

- `mailService.send({ to, subject, text?, html?, from? })` — send raw email
- `mailService.sendTemplate({ to, template, variables?, from? })` — render and send template-based email
- `mailService.sendVerificationCode(to, code, expiresInMinutes?)` — send verification code (default 5 minutes)

Example:

```typescript
// Send raw email
await mailService.send({
  to: 'user@example.com',
  subject: 'Hello',
  text: 'This is a test email',
});

// Send templated email
await mailService.sendTemplate({
  to: 'user@example.com',
  template: 'welcome',
  variables: { userName: 'John' },
});

// Send verification code
await mailService.sendVerificationCode('user@example.com', '123456', 10);
```

If mail is disabled, all methods throw `BadRequestException('Mail is not configured')`.
